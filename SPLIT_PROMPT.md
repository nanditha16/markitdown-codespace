# Split-Prompt Architecture for Stage 0/1 (Variant Ranking)

## What changed and why

Previously `variant_rank.sh` bundled ALL resume variants AND the JD into a
single monolithic prompt file (`prompts/variant_rank_prompt.txt`) on every
run. With a 15-20 variant bank (each 1,500–2,500 words), this regenerated
30–50KB of identical variant content every time a new JD was evaluated.

**Now Stage 0/1 uses two separate files:**

| File | Generated by | When | Purpose |
|---|---|---|---|
| `prompts/variant_bank.txt` | `build_variant_bank.sh` | Once (or when variants change) | Role instructions + all variant content |
| `prompts/jd_current.txt` | `variant_rank.sh` | Every JD run | JD text only |

Both files are uploaded together to Claude.ai in the same conversation.
`variant_bank.txt` instructs Claude to read the JD from the separately
attached `jd_current.txt`.

**Benefits:**
- Re-running on a new JD only writes ~1KB instead of 30–50KB
- The bank file is inspectable and version-controllable separately
- Staleness detection warns if any variant is newer than the bank

## Updated Stage 0/1 Workflow

### Command line

```bash
# ── FIRST TIME (or when variants change) ──────────────────────────────────
# Build the static variant bank (only need to redo if you add/edit variants):
./scripts/build_variant_bank.sh
# → prompts/variant_bank.txt  (30-50KB, keep this file across all JD runs)

# ── EACH NEW JD ───────────────────────────────────────────────────────────
# Generate the JD-only file:
./scripts/variant_rank.sh "input/other/JD5.txt"
# → prompts/jd_current.txt  (~1KB)

# Upload BOTH to Claude.ai in one conversation:
#   1. prompts/variant_bank.txt
#   2. prompts/jd_current.txt
# Paste Claude's response into:
#   prompts/JD_Analysis/JD5/variant_rank_prompt_response.txt

# ── AFTER ADDING/EDITING VARIANTS ─────────────────────────────────────────
./scripts/build_variant_bank.sh --rebuild
# Then run variant_rank.sh as normal for the next JD
```

### Staleness detection

`build_variant_bank.sh` compares the bank file's modification time against
every variant in `output/resume/`. If any variant is newer than the bank:

- **When called standalone:** prints a warning and rebuilds automatically
- **When called by `variant_rank.sh`:** prints a warning; does NOT auto-rebuild
  (non-blocking, since the user may have intentionally excluded a new variant)
- **Pass `--rebuild`** to force rebuild in either case

### Force rebuild

```bash
./scripts/build_variant_bank.sh --rebuild
```

### Web UI

The Stage 1 panel in the web UI now shows:

- **Variant bank status**: exists / missing / stale (with which variant is newer)
- **"Build Bank" button**: runs `build_variant_bank.sh` with live streaming output
- **"Rebuild Bank" button**: forces rebuild even if bank is current
- **"Download variant_bank.txt"** button: download the static bank file
- **"Generate JD File"** button: runs `variant_rank.sh` for the selected JD
- **"Download jd_current.txt"** button: download the JD-only file

Upload both downloads to Claude.ai together.

## Scripts reference (updated)

| Script | What it does | When to run |
|---|---|---|
| `build_variant_bank.sh` | Generates `prompts/variant_bank.txt` from all variants in `output/resume/` | Once; again after adding/editing variants |
| `variant_rank.sh <jd>` | Checks bank freshness, generates `prompts/jd_current.txt` | Every JD evaluation |
| `ats_workflow.sh` | Full orchestrator; calls `build_variant_bank.sh` automatically | Full single-JD run |

`build_variant_bank.sh` flags:

```bash
./scripts/build_variant_bank.sh                  # build if missing/stale; skip if current
./scripts/build_variant_bank.sh --rebuild         # force rebuild unconditionally
./scripts/build_variant_bank.sh "custom/dir"      # use non-default variant directory
./scripts/build_variant_bank.sh --rebuild "dir"   # force + custom dir
```

## Manual testing steps

Use these to verify the split-prompt changes work before using the web UI:

```bash
# 1. Confirm no variant bank exists yet (clean state):
ls prompts/variant_bank.txt 2>/dev/null || echo "bank not found — expected"

# 2. Build the variant bank:
./scripts/build_variant_bank.sh
# Expected: "✅ Variant bank saved to prompts/variant_bank.txt"

# 3. Re-run — should skip because bank is current:
./scripts/build_variant_bank.sh
# Expected: "✅ prompts/variant_bank.txt is up to date"

# 4. Generate a JD file:
./scripts/variant_rank.sh "input/other/JD1.txt"
# Expected: "✅ JD file written to prompts/jd_current.txt"
# Expected: prints upload instructions for BOTH files

# 5. Simulate stale bank by touching a variant:
touch output/resume/Your_Resume_SomeCompany.md
./scripts/build_variant_bank.sh
# Expected: "⚠️  Variant '...' is newer than variant_bank.txt. Rebuilding..."

# 6. Test --rebuild flag:
./scripts/build_variant_bank.sh --rebuild
# Expected: rebuilds unconditionally, no staleness check

# 7. Test variant_rank.sh staleness warning (non-blocking):
touch output/resume/Your_Resume_SomeCompany.md
./scripts/variant_rank.sh "input/other/JD1.txt"
# Expected: "⚠️  WARNING: '...' is newer than variant_bank.txt."
#           but still generates jd_current.txt

# 8. Verify the bank does NOT contain JD text:
grep -c "JOB DESCRIPTION" prompts/variant_bank.txt
# Expected: 0  (the bank has no embedded JD)

# 9. Verify jd_current.txt contains only the JD:
grep -c "VARIANT:" prompts/jd_current.txt
# Expected: 0  (no variant content in the JD file)

# 10. Verify bank contains upload instructions:
head -10 prompts/variant_bank.txt
# Expected: shows the "IMPORTANT — HOW TO USE THIS FILE" header
```
